# 연차/휴무 신청 개편 설계 (2026-06-17)

## 배경
현재 직원 신청 화면에서 **휴무(normal)**는 캘린더로 날짜를 고르고(세션당 최대 3일), **연차(annual)**는 날짜 없이 "그 달 1일 더미 + 월 1건"으로만 신청된다. 운영자는 연차를 "본인 잔여 한도 내에서 며칠이든" 쓰게 하고 싶어 한다.

## 확정 규칙
- **R1 — 월 합산 3일 상한**: 한 직원이 한 달에 신청 가능한 휴무+연차 합계는 최대 3일. **월 단위**로 카운트(이미 제출한 비거절 요청 + 이번에 새로 고른 것 합산).
- **R2 — 잔여 연차 상한**: 연차 신규 선택 ≤ `remaining − (현재 pending 연차 요청 수)`. `remaining = lifetime accrued − lifetime used(ledger)`.
- **R3 — 같은 날 1종류**: 한 날짜는 휴무 또는 연차 중 하나만.
- **R4 — 연차 날짜화**: 연차도 휴무와 동일한 캘린더에서 날짜 선택(여러 일). "그 달 1일 더미", "월 1건" 폐기.
- **R5 — ledger 연동**: 연차 승인 시 해당 날짜로 ledger `usage` 1일 기록, 삭제/취소 시 해당 날짜 usage 삭제(기존과 동일하되 실제 날짜 기준).
- **R6 — 스케줄 반영**: 승인된 연차는 실제 날짜로 스케줄·관리자 휴무 캘린더·연속근무 경고에 반영.

### 예시
- 잔여 5일: 연차 3일 / 연차 2일+휴무 1일 / 휴무 3일 (합계 ≤3). 연차 4일 ❌(월 3일 초과).
- 잔여 1일: 연차 1일+휴무 2일 ✅. 연차 2일 ❌(잔여 부족).

## 변경 대상
- `js/employee-request.js` — 핵심. 단일 캘린더 + 모드(휴무/연차) 토글로 통합. `pickDate`/`setType`/`submitDayOffRequest` 재작성. 날짜→유형 매핑(`Map`)으로 상태 관리. R1/R2/R3 픽·제출 단계에서 강제.
- `js/admin-requests.js` — 연차 승인이 실제 날짜로 ledger 기록(이미 date 인자 전달됨, 더미 제거로 자연 해결). 잔여 초과 승인 시 confirm 경고. `buildOffCalendar`에 연차(annual) 포함.
- `js/db.js` — 직원의 pending 연차 수를 구하는 헬퍼 추가(전 기간 스코프). 잔여 초과 방지용.
- (선택) 캡 산정 순수 함수 분리 + jest 테스트.

## 비범위(out of scope)
- 휴무 3일 캡 숫자 변경(그대로 3).
- 근속 가산휴가, conditions 테이블 정합화(max_normal_offs=8 vs UI=3)는 별도 건으로 보고만.
- 서버사이드 검증/RLS 강화(소규모 내부용 전제 유지).

## 보안/정합성 주의
- 모든 검증은 클라이언트 사이드(RLS allow-all). 동시 제출/경합은 막지 못함 → 관리자 승인 단계의 잔여 초과 confirm으로 2차 방어.
- 날짜 문자열은 `YYYY-MM-DD` 로컬 기준 유지.
